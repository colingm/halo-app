### Human Verification Required

#### 1. End-to-end signup wizard completion in a real browser

**Test:** Open the dev server, navigate to /signup, complete all four steps with valid input, click "Create account."
**Expected:**
- Each step lives at its own URL (`/signup`, `/signup/details`, `/signup/company`, `/signup/preferences`)
- After Step 1's Continue, DevTools sessionStorage panel shows `halo:v1:signup:draft` value contains `step1` slice WITHOUT a `password` field
- Stepper visual updates to match current step
- After clicking "Create account" on Step 4: redirected to `/app`; localStorage has new `halo:v1:visitors` + `halo:v1:workspaces` + `halo:v1:session` entries; `halo:v1:signup:draft` is removed from sessionStorage
- The persisted Visitor's `passwordHash` is a 64-char hex string, NOT the plaintext password
**Why human:** The CR-02 plaintext-password-in-memory pattern is the most behaviorally novel piece of Phase 2 and the only place where a manual DevTools inspection is the right verification. Smoke tests confirm the code paths individually, but the end-to-end claim ("plaintext never reaches sessionStorage") needs a real browser session.

#### 2. Refresh-mid-wizard password re-entry behavior

**Test:** On Step 2, 3, or 4 with prior steps completed, refresh the browser.
**Expected:** The in-memory plaintext password is gone; the user must re-enter Step 1 (because completion requires `getWizardPassword()` to return non-null). On Step 4, clicking "Create account" without the in-memory password should trigger the generic failure Alert.
