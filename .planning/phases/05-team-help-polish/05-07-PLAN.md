<objective>
Resolve Gaps 1 + 4 from `.planning/phases/05-team-help-polish/05-UAT.md` — the cold-start seed reconciliation regression. Replace the seedAll coordinator's single global `meta.seededAt` gate with a per-domain seeding ledger (`meta.seededDomains: { tasks?: string; teammates?: string }`) so each domain seeder is checked independently. Legacy records (pre-Phase-5 installs where the old `tasksSeed.ts` tail stamped `seededAt` before teammate seeding existed) are interpreted as `{ tasks: <legacy stamp> }` at read time — leaving `teammates` unset so the next boot runs the teammate seeder only, preserving existing task data.

Purpose: One bug surface causes both Gap 1 (Test 1 — "no seeded data on the Team page") and Gap 4 (Test 7 — "no Owner row on the Team page") per the diagnoses in `.planning/debug/05-team-seed-missing-cold-start.md` and `.planning/debug/05-team-owner-row-missing.md`. The seeder logic itself is correct (Test 10 reset+re-seed proves this) — only the coordinator-level idempotency gate is wrong. This plan surgically replaces that gate without bumping SCHEMA_VERSION or touching migrations.ts.

Planner choice (Option A over Option B): per-domain seeding ledger preserves existing task data, future-proofs additional seed domains, and the legacy backward-compat is a one-line read-side mapping. Option B (v1->v2 migration nulling seededAt) is blunter — it would force re-seeding tasks on every legacy install, losing any task mutations users had made. CLAUDE.md's "fabricated demo data only" framing allows Option B in principle, but Option A is cleaner and matches the architectural direction (Phase 5 D-11 already calls for "per-domain seeders with a single coordinator" — the new ledger completes that vision). D-12 (single meta.seededAt gate) is the decision this plan deliberately revises; the rationale is documented in the seedAll.ts file-header docblock for traceability.

Output: 1 modified schema (additive optional field, backward-compat), 1 modified coordinator (new per-domain gate logic + legacy reconciliation), 2 modified seeders (gate updates symmetric to new ledger). Zero new files. Zero SCHEMA_VERSION change. Zero migration entries.
</objective>
